      Integrated system --- RASR！

      According to the existing data analysis process, tools, and execution code, the integrated pipeline for the whole process of data parallel batch download, inspection, and processing has been completed and is currently uploaded to github (https://github.com/clsteam/RSAR). Specifically include environmental testing, data reading, text analysis, organization of sample materials, parameter settings, log output, results verification, and more.

      What are the functions of RASR?

  1. Environmental testing

           Java, python, Samtools, Bgzip, and other dependent environments are the premise of the program's operation. Therefore, it is first to check if the environment is perfect. This can be verified by a function check_dependency(), where an environment must be directly reported as an error and a non-essential environment is missing as an output warning.
           E.g:
                 If Tophat2 is missing, it is directly reported as "External dependency 'tophat2' not installed."
                 If formatdb(blast) is missing, then the external dependency 'formatdb ' not installed.fusion search will not be running.
  2. Data reading

           The input data forms are various and can be selected according to the user's needs:
                 1) Overall input in the form of a folder (suitable for large amount of data and single data format)
                       After the parameter --data, enter the target folder. The folder contains all the original data. The format of the original data can be fastq, fastq.gz, and sra. Both single-end and double-end data can be used.
                 2) Input in the form of a file directory (suitable for a small amount of data and a messy data format)
                 After the parameter -data is input to the target file, the file contains the address of the data to be analyzed. The format of the original data may be fastq, fastq.gz, sra, and both single-ended and double-end data may be used.
           The reference format is as follows:
                 #genome annotations bed blacklist haplo_count_blacklist
                 #ref PE a1.fastq,a2.fastq
                 #ref PE b1.fastq,b2.fastq
                 #lab SE a.fastq
                 #lab SE b.fastq
                 #lab SRA c.sra
  3. Text Analysis

           The results of the quality inspection software FastQC need to manually determine whether the quality is qualified, whether the GC content is up to standard, whether there is overexpression and adpter, primer sequence. Automation, on the other hand, can only be judged and artificial links eliminated. First read the fastqc result file data.txt, view the GC content, the overall length and quality of the sequence, etc., then remove the noisy sequence and the detected adapter sequence by the overall ATGC content.
    Note: Single position (A/T) - (G/C) > 10% sequence removal.

  4. Sample Material Organization

           This process includes the establishment of a parallel pool, the creation of folders, the distribution of analysis results, the generation and deletion of intermediate results, so that the overall analysis can be organized and completed independently.

  5. parameter settings

           Each tool has its own parameter settings, the use of conditions and purposes are different, some parameters can be the default, some parameters need to match up and down, some parameters must be provided by the user, so here set a different type of parameters. E.g:
           The --paired-end and parameters in Phaser: need to match up and down according to the situation, and -mapq, -baseq parameters are optional default (we default to 0), but some cases require the user to give specific values, otherwise there is no result Output.

  6. Log output
           The output types are divided into 5 categories:
                  Fatal_error() Fatal error, exit directly, stop running, output error
                  Print_warning() warning tips
                  Mes_print() output progress
                  Print_debug() prints out specific information in a progress
                  Must_print() must output information (eg file format, version information, etc.)

           In order to avoid cross-interference between the original tool output and the pipeline output, all outputs of the tool advance dev/null and replace it with the output of the pipeline. Because the program is parallel, in order to distinguish the progress of each sample analysis, Each step is marked with @mark.

  7. Test of results

           Because there is no guarantee that the user will not experience unexpected program interruption during use, the result added in the result output step is generated. Whether the script is complete or not, the next time the program is submitted again, it can be judged whether the step has been executed. Skip if it is already running, otherwise run.
           And in order to avoid multiple submissions of the same task, the same task name set by the pipeline cannot be repeatedly submitted.
